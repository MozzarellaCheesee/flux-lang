## Задача лексера

Лексер (токенизатор) — первый этап компилятора. Его задача: преобразовать плоскую строку исходного кода в последовательность **токенов** — смысловых единиц языка.

```
"let x: int32 = 42;"
    →
[KW_LET] [IDENT "x"] [COLON] [KW_INT32] [EQ] [LIT_INT "42"] [SEMICOLON] [EOF]
```

## Структура токена

```cpp
// include/flux/lexer/token.h
struct Token {
    TokenKind      kind;    // вид токена
    std::string_view lexeme; // срез исходной строки (без копирования)
    SourceLocation loc;     // файл, строка, столбец
};
```

### Почему `string_view` вместо `string`?

`std::string_view` — невладеющий срез чужой строки. Это позволяет избежать миллионов копирований подстрок при токенизации большого файла. Недостаток: исходная строка (`std::string source`) должна пережить все токены. В препроцессоре это решается через `source_store_` — вектор, который хранит строки импортированных файлов.

## Виды токенов (TokenKind)

Полный список определён в `include/flux/lexer/token_kind.h`.

### Ключевые слова

| Токен | Лексема |
|-------|---------|
| `KW_FNC` | `fnc` |
| `KW_LET` | `let` |
| `KW_RETURN` | `return` |
| `KW_IF` | `if` |
| `KW_ELSE` | `else` |
| `KW_FOR` | `for` |
| `KW_WHILE` | `while` |
| `KW_BREAK` | `break` |
| `KW_CONTINUE` | `continue` |
| `KW_TRUE` | `true` |
| `KW_FALSE` | `false` |
| `KW_MATCH` | `match` |
| `KW_PUB` | `pub` |
| `KW_STRUCT` | `struct` |
| `KW_CLASS` | `class` |
| `KW_IMPL` | `impl` |
| `KW_SELF` | `self` |
| `KW_EXTERN` | `extern` |

### Типы как ключевые слова

| Токен | Лексема | C++ |
|-------|---------|-----|
| `KW_BOOL` | `bool` | `bool` |
| `KW_INT8` .. `KW_INT128` | `int8` .. `int128` | `int8_t` .. `__int128` |
| `KW_UINT8` .. `KW_UINT128` | `uint8` .. `uint128` | `uint8_t` .. `unsigned __int128` |
| `KW_FLOAT` | `float` | `float` |
| `KW_DOUBLE` | `double` | `double` |
| `KW_STR` | `str` | `std::string_view` |
| `KW_STRING` | `string` | `std::string` |
| `KW_ISIZE_T` | `isize_t` | `ptrdiff_t` |
| `KW_USIZE_T` | `usize_t` | `size_t` |

**Почему типы — ключевые слова?** Это упрощает парсер типов: не нужно отдельно проверять, является ли идентификатор именем встроенного типа. Парсер просто смотрит на вид токена.

### Литералы

| Токен | Пример |
|-------|--------|
| `LIT_INT` | `42`, `0`, `100` |
| `LIT_FLOAT` | `3.14`, `0.5` |
| `LIT_STRING` | `"hello"` |

### Операторы (выборка)

| Токен | Символ | Описание |
|-------|--------|----------|
| `PLUS`, `MINUS`, `STAR`, `SLASH`, `PERCENT` | `+`, `-`, `*`, `/`, `%` | Арифметика |
| `EQ_EQ`, `BANG_EQ` | `==`, `!=` | Сравнение |
| `LT`, `GT`, `LT_EQ`, `GT_EQ` | `<`, `>`, `<=`, `>=` | Сравнение |
| `AMP_AMP`, `PIPE_PIPE` | `&&`, `\|\|` | Логика |
| `AMP`, `PIPE`, `CARET`, `TILDE` | `&`, `\|`, `^`, `~` | Побитовые |
| `LSHIFT`, `RSHIFT` | `<<`, `>>` | Сдвиги |
| `PLUS_EQ`, `MINUS_EQ` и др. | `+=`, `-=` и др. | Составное присваивание |
| `PLUS_PLUS`, `MINUS_MINUS` | `++`, `--` | Инкремент/декремент |
| `ARROW` | `->` | Тип возврата функции |
| `FAT_ARROW` | `=>` | Ветка match |

### Препроцессорные токены

| Токен | Синтаксис | Описание |
|-------|-----------|----------|
| `PP_IMPORT` | `#import "path"` | Импорт файла |
| `PP_IMPORT_SYS` | `#import <module>` | Импорт stdlib |

Для `PP_IMPORT_SYS` лексема — **только имя модуля** (без `<` и `>`). Это делает обработку в препроцессоре простой: `tok.lexeme == "io"`.

## Алгоритм токенизации

Метод `next_token()` реализует ручной лексер:

```cpp
Token Lexer::next_token() {
    size_t start = cursor_;
    char c = advance();   // берём следующий символ

    switch (c) {
        case '(': return make_token(TokenKind::LPAREN, start);
        case '+':
            if (peek() == '+') { advance(); return make_token(PLUS_PLUS, start); }
            if (peek() == '=') { advance(); return make_token(PLUS_EQ, start); }
            return make_token(PLUS, start);
        // ... и так для каждого символа
        case '#': return lex_hash_directive(start);
        case '"': return lex_string(start);
        default:
            if (std::isdigit(c)) return lex_number(start);
            if (std::isalpha(c) || c == '_') return lex_ident_or_keyword(start);
    }
}
```

**Почему ручной лексер, а не генератор (flex, re2c)?** Ручной лексер проще отлаживать, а производительности более чем достаточно для небольших программ.

## Пропуск пробелов и комментариев

Перед каждым токеном вызывается `skip_whitespace_and_comments()`:

```cpp
// Однострочный комментарий
while (!at_end() && peek() != '\n') advance();

// Многострочный /* ... */
advance(); advance(); // съедаем /*
while (!at_end()) {
    if (peek() == '*' && peek_next() == '/') {
        advance(); advance(); break;
    }
    advance();
}
```

## Распознавание ключевых слов

После лексирования идентификатора проверяется таблица `KEYWORDS`:

```cpp
// include/flux/lexer/lexer.h
static const std::unordered_map<std::string_view, TokenKind> KEYWORDS = {
    {"fnc",    TokenKind::KW_FNC},
    {"let",    TokenKind::KW_LET},
    {"extern", TokenKind::KW_EXTERN},
    // ...
};

Token Lexer::lex_ident_or_keyword(size_t start) {
    while (!at_end() && (std::isalnum(peek()) || peek() == '_')) advance();
    std::string_view lexeme = source_.substr(start, cursor_ - start);
    auto it = KEYWORDS.find(lexeme);
    TokenKind kind = (it != KEYWORDS.end()) ? it->second : TokenKind::IDENT;
    return make_token(kind, start);
}
```

Поиск в `unordered_map<string_view>` — O(1) по хешу.

## Обработка `#import`

`lex_hash_directive()` — специальный метод для директив препроцессора:

```cpp
Token Lexer::lex_hash_directive(size_t start) {
    while (!at_end() && peek() == ' ') advance();  // пропускаем пробелы
    // читаем имя директивы
    size_t kw_start = cursor_;
    while (!at_end() && std::isalpha(peek())) advance();
    std::string_view directive = source_.substr(kw_start, cursor_ - kw_start);

    if (directive == "import") {
        while (!at_end() && peek() == ' ') advance();
        if (!at_end() && peek() == '<') {
            advance();  // '<'
            size_t mod_start = cursor_;
            while (!at_end() && peek() != '>') advance();
            // создаём токен PP_IMPORT_SYS с лексемой = имя модуля
            Token tok;
            tok.kind   = TokenKind::PP_IMPORT_SYS;
            tok.lexeme = source_.substr(mod_start, cursor_ - mod_start);
            tok.loc    = {filepath_, line_, col_};
            advance();  // '>'
            return tok;
        }
        return make_token(TokenKind::PP_IMPORT, start);
    }
    // неизвестная директива — ошибка
}
```

`PP_IMPORT_SYS` создаётся **вручную** (не через `make_token`), потому что нужна нестандартная лексема — только имя модуля, без `#import <`.

## SourceLocation

```cpp
// include/flux/common/source_location.h
struct SourceLocation {
    std::string filepath;
    uint32_t    line;
    uint32_t    col;
};
```

Лексер ведёт счётчики `line_` и `col_`, обновляя их при каждом `advance()`:
```cpp
char Lexer::advance() {
    char c = source_[cursor_++];
    if (c == '\n') { line_++; col_ = 1; }
    else           { col_++; }
    return c;
}
```
