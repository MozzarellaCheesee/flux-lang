## Общая схема

Компиляция программы на Flux проходит через 9 последовательных шагов:

```
 ┌────────────────────────────────────────────────────────┐
 │                     fluxc.exe                          │
 │                                                        │
 │  source.flx                                            │
 │      │                                                 │
 │      ▼                                                 │
 │  [1] Lexer         → std::vector<Token>                │
 │      │                                                 │
 │      ▼                                                 │
 │  [2] Preprocessor  → std::vector<Token> (раскрытый)    │
 │      │                                                 │
 │      ▼                                                 │
 │  [3] Parser        → unique_ptr<ProgramNode>           │
 │      │                                                 │
 │      ▼                                                 │
 │  [4] SemanticAnalyzer  (изменяет ProgramNode in-place) │
 │      │                                                 │
 │      ▼                                                 │
 │  [5] CodeGen       → std::string (C++ код)             │
 │      │                                                 │
 │      ▼                                                 │
 │  [6] Запись .cpp в /tmp                                │
 │      │                                                 │
 └──────┼─────────────────────────────────────────────────┘
        │
        ▼
 [7] g++ / clang++ / cl  (внешний процесс)
        │
        ▼
     program.exe
        │
        ▼
 [8] Очистка временного .cpp
 [9] Вывод "Built: program.exe"
```

---

## Шаг 1 — Лексер

**Входные данные:** строка с исходным кодом (`std::string_view`)
**Выходные данные:** `std::vector<Token>`
**Файлы:** `src/lexer/lexer.cpp`, `include/flux/lexer/`

Лексер читает символы по одному и группирует их в токены. Каждый токен хранит:
- `kind` — вид токена (перечисление `TokenKind`)
- `lexeme` — срез исходной строки (`std::string_view` — без копирования!)
- `loc` — позиция: файл, строка, столбец

Решение использовать `string_view` для lexeme вместо копирования строк — сознательная оптимизация. Это означает, что исходная строка должна жить дольше токенов.

**Подробнее:** [[lexer]]

---

## Шаг 2 — Препроцессор

**Входные данные:** токены одного файла
**Выходные данные:** объединённый поток токенов со всеми импортами вставленными рекурсивно
**Файлы:** `src/preprocessor/preprocessor.cpp`

При встрече `PP_IMPORT` ("path") препроцессор:
1. Читает файл с диска
2. Лексирует его
3. Рекурсивно обрабатывает (для вложенных `#import`)
4. Вставляет токены в поток

При встрече `PP_IMPORT_SYS` (`<module>`) — ищет модуль во встроенной таблице.

**Защита от циклов:** множество `visited_` с каноническими путями. Для stdlib-модулей ключ имеет вид `__stdlib_io`.

**Подробнее:** [[Препроцессор]]

---

## Шаг 3 — Парсер

**Входные данные:** плоский поток токенов
**Выходные данные:** `std::unique_ptr<ProgramNode>` — корень AST
**Файлы:** `src/parser/parser.cpp`

Метод: **рекурсивный спуск** для объявлений и операторов, **алгоритм Пратта** (binding power) для выражений.

Если разбор не удался — диагностика через `DiagEngine`. Парсер старается **восстановиться** после ошибки (съедает неожиданный токен и продолжает), чтобы показать сразу несколько ошибок.

**Подробнее:** [[parser]]

---

## Шаг 4 — Семантический анализатор

**Входные данные:** `ProgramNode&`
**Выходные данные:** диагностика (ошибки/предупреждения); узлы AST могут обогащаться (ImplicitCastExpr)
**Файлы:** `src/sema/sema.cpp`

Два прохода:
1. **Первый проход** — регистрирует все функции в `func_table_` (чтобы функции могли вызывать друг друга до своего объявления)
2. **Второй проход** — полная проверка каждого узла

**Подробнее:** [[Семантический анализатор]]

---

## Шаг 5 — Кодогенератор

**Входные данные:** `ProgramNode&`
**Выходные данные:** `std::string` — полный C++ исходник
**Файлы:** `src/codegen/codegen.cpp`

Сначала выводит `flux_runtime` (~200 строк C++), затем обходит AST и генерирует соответствующий C++ код.

**Подробнее:** [[Кодогенератор]]

## Шаги 6–9 — Компиляция C++ и очистка

Временный файл создаётся в `std::filesystem::temp_directory_path()` с уникальным именем `<stem>_<pid>.cpp`. Это гарантирует, что параллельно запущенные компиляции не конфликтуют между собой.

Компилятор выбирается автоматически в порядке предпочтения: `g++` → `clang++` → `cl`. Явно задать можно флагом `--compiler gcc|clang|msvc`.

```cpp
// main.cpp — логика выбора компилятора
static std::optional<std::string> resolve_compiler(CompilerKind kind) {
    // Auto: g++ → clang++ → cl
}
```

Команда компиляции:
- GCC/Clang: `g++ -std=c++20 -o output.exe input.cpp`
- MSVC: `cl /std:c++20 /EHsc /Fe:output.exe input.cpp`

После успешной компиляции временный `.cpp` удаляется. При `--emit-cpp` он перемещается рядом с бинарником.

## Обработка ошибок в конвейере

Каждый шаг проверяет `diag.has_errors()` перед передачей управления следующему:

```cpp
// main.cpp
if (diag.has_errors()) { diag.print_all(); return 1; }  // после лексера
if (diag.has_errors()) { diag.print_all(); return 2; }  // после препроцессора
if (diag.has_errors()) { diag.print_all(); return 3; }  // после парсера
if (diag.has_errors()) { diag.print_all(); return 5; }  // после семантики
```

Коды возврата отличаются намеренно — по коду можно понять на каком этапе упала компиляция.
