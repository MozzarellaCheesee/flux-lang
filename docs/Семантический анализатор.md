## Задача

Семантический анализатор — третий этап после парсера. Парсер проверяет **синтаксис** (правильно ли расставлены скобки), семантический анализатор проверяет **смысл** (правильны ли типы, объявлены ли все переменные).

Примеры семантических ошибок:
```flux
let x: int32 = "hello";      // ошибка: нельзя присвоить &str к int32
let y = z + 1;               // ошибка: переменная z не объявлена
return 3.14;                 // ошибка: функция должна вернуть int32
```

## Два прохода

```cpp
void SemanticAnalyzer::visit(ProgramNode& node) {
    // Проход 1: регистрируем ВСЕ функции
    for (auto& decl : node.decls) {
        if (auto* fn = dynamic_cast<FuncDecl*>(decl.get()))
            register_func(*fn, "");
        if (auto* impl = dynamic_cast<ImplDecl*>(decl.get()))
            for (auto& m : impl->methods)
                register_func(*m, impl->target);
        if (auto* cls = dynamic_cast<ClassDecl*>(decl.get()))
            for (auto& m : cls->methods)
                register_func(*m, cls->name);
    }

    // Проход 2: полный анализ
    for (auto& decl : node.decls)
        if (decl) decl->accept(*this);

    check_main_entry_point();
}
```

**Зачем два прохода?** В Flux функции могут вызывать друг друга до своего объявления:
```flux
fnc foo() -> int32 { return bar(); }  // bar ещё не объявлена!
fnc bar() -> int32 { return 42; }
```
Без предварительной регистрации `bar` была бы "неизвестна" при разборе `foo`.

## Таблица функций (FuncTable)

```cpp
// include/flux/sema/sema.h
struct FuncSignature {
    std::string              name;
    std::vector<std::string> param_types;   // ["int32", "double"]
    std::string              return_type;    // "bool"
    FuncDecl*                decl;           // указатель на AST-ноду
    bool                     is_pub;
    std::string              owner;          // "" для свободных, "Point" для методов
};

using OverloadSet = std::vector<FuncSignature>;
using FuncTable   = std::unordered_map<std::string, OverloadSet>;
```

Один ключ в `FuncTable` может содержать несколько перегрузок:
```flux
fnc max(a: int32, b: int32) -> int32 { ... }
fnc max(a: double, b: double) -> double { ... }
// func_table_["max"] = [{int32,int32→int32}, {double,double→double}]
```

## Разрешение перегрузок

```cpp
const FuncSignature* SemanticAnalyzer::resolve_overload(
    const std::string& name,
    const std::vector<std::string>& arg_types,
    const SourceLocation& loc)
{
    auto it = func_table_.find(name);
    if (it == func_table_.end()) {
        diag_.emit(Error, loc, "Undefined function '" + name + "'");
        return nullptr;
    }

    // Точное совпадение по типам аргументов (с учётом is_assign_compatible)
    for (const auto& sig : it->second) {
        if (sig.param_types.size() != arg_types.size()) continue;
        bool match = true;
        for (size_t i = 0; i < arg_types.size(); ++i)
            if (!is_assign_compatible(arg_types[i], sig.param_types[i]))
                { match = false; break; }
        if (match) return &sig;
    }

    // Ни одна перегрузка не подошла → подробное сообщение об ошибке
    // с перечислением доступных сигнатур
    diag_.emit(Error, loc, "No matching overload for '" + name + "' ...");
    return nullptr;
}
```

## Области видимости

Анализатор поддерживает стек областей видимости — каждый `{}` создаёт новый scope:

```cpp
// Стек: каждый элемент — map имя→тип
std::vector<std::unordered_map<std::string, std::string>> scopes_;

void push_scope() { scopes_.emplace_back(); }
void pop_scope()  { scopes_.pop_back(); }

// Поиск снизу вверх (от внутреннего к внешнему)
std::string lookup_var(const std::string& name) const {
    for (auto it = scopes_.rbegin(); it != scopes_.rend(); ++it) {
        auto found = it->find(name);
        if (found != it->end()) return found->second;
    }
    return "";  // не найдено
}
```

При объявлении переменной, которая уже есть в текущем scope, выдаётся предупреждение о "shadowing":
```
warning: Variable 'x' shadows previous declaration
```

## Проверка типов

### eval_type

Центральная функция — `eval_type(ASTNode&)`. Она запускает visitor на узле и возвращает тип последнего вычисленного выражения:

```cpp
std::string eval_type(ASTNode& node) {
    node.accept(*this);
    return last_type_;
}
```

Каждый `visit` для выражений устанавливает `last_type_`:
```cpp
void visit(IntLiteral&)   override { last_type_ = "int32"; }
void visit(FloatLiteral&) override { last_type_ = "double"; }
void visit(BoolLiteral&)  override { last_type_ = "bool"; }
void visit(StrLiteral&)   override { last_type_ = "&str"; }
```

### is_assign_compatible

Проверяет, можно ли присвоить значение типа `from` переменной типа `to`:

```cpp
static bool is_assign_compatible(const std::string& from, const std::string& to) {
    if (from == to) return true;

    // Любые числовые типы совместимы между собой
    if (is_numeric(from) && is_numeric(to)) return true;

    // &str / str / string — строковые типы взаимно совместимы
    auto is_str = [](const std::string& t) {
        return t == "string" || t == "&str" || t == "str";
    };
    if (is_str(from) && is_str(to)) return true;

    // Result<T,E> с wildcard (от Ok/Err конструкторов)
    if (from.starts_with("Result<") && to.starts_with("Result<"))
        if (from.find('_') != npos || to.find('_') != npos)
            return true;

    return false;
}
```

**Почему числовые типы взаимно совместимы?** В Flux нет явного приведения типов (`as`). Компилятор позволяет смешивать числовые типы — при генерации C++ компилятор C++ выполнит необходимое неявное приведение. Для учебного языка это приемлемо.

**Почему `&str` совместим с `string`?** В C++ `std::string s = "literal"` работает через неявное преобразование. Мы хотим тот же опыт в Flux. Реализация: тип строкового литерала `&str` (string_view), но при присваивании в `string`-переменную C++ выполнит конструктор `std::string(string_view)`.

### wider_type — расширение типов в арифметике

```cpp
// int32 + double → double
// int8 + int32 → int32
static std::string wider_type(const std::string& a, const std::string& b) {
    int ra = numeric_rank(a), rb = numeric_rank(b);
    return ra >= rb ? a : b;
}

static int numeric_rank(const std::string& t) {
    if (t == "double") return 100;
    if (t == "float")  return 90;
    if (t == "int128" || t == "uint128") return 70;
    if (t == "int64"  || t == "uint64")  return 60;
    if (t == "int32"  || t == "uint32")  return 50;
    // ...
}
```

## Проверка операторов

```cpp
void visit(BinaryExpr& node) {
    std::string lhs = eval_type(*node.lhs);
    std::string rhs = eval_type(*node.rhs);

    // == != < > <= >= → всегда bool
    if (cmp_ops.count(node.op)) {
        if (!is_implicitly_convertible(lhs, rhs) && !is_implicitly_convertible(rhs, lhs))
            diag_.emit(Error, node.loc, "Cannot compare '" + lhs + "' with '" + rhs + "'");
        last_type_ = "bool";
        return;
    }

    // && || → оба должны быть bool
    if (node.op == "&&" || node.op == "||") {
        if (lhs != "bool") diag_.emit(Error, ...);
        if (rhs != "bool") diag_.emit(Error, ...);
        last_type_ = "bool";
        return;
    }

    // + - * / % → оба числовые, результат — более широкий
    if (is_numeric(lhs) && is_numeric(rhs)) {
        last_type_ = wider_type(lhs, rhs);
        return;
    }
}
```

## Проверка match

```cpp
void visit(MatchExpr& node) {
    std::string subject_type = eval_type(*node.subject);

    for (auto& arm : node.arms) {
        push_scope();
        // WildcardPattern → OK
        // LiteralPattern  → проверяем тип литерала
        // ConstructorPattern (Ok/Err) → проверяем что subject это Result<...>
        arm->body->accept(*this);
        pop_scope();
    }
    last_type_ = "unknown"; // тип match-выражения пока не определяется
}
```

## Проверка точки входа main

```cpp
void check_main_entry_point() {
    if (!func_table_.count("main")) {
        diag_.emit(Error, {}, "No entry point: 'main' function not found");
        return;
    }

    const auto& sig = func_table_["main"][0];

    // Допустимые возвращаемые типы
    bool valid = sig.return_type == "int32" ||
                 sig.return_type == "Result<int32, string>";
    if (!valid) diag_.emit(Error, ...);

    // Допустимые параметры: () или (argc: int32, argv: &&str)
    if (sig.param_types.size() == 0) return;  // OK
    if (sig.param_types.size() == 2) {
        // проверяем argc: int32, argv: &&str
        return;
    }
    diag_.emit(Error, ..., "'main' must have 0 or 2 parameters");
}
```

## Таблица типов (TypeTable)

Параллельно с `func_table_` ведётся `type_table_` — реестр всех объявленных структур и классов:

```cpp
std::unordered_map<std::string, TypeInfo> type_table_;

struct TypeInfo {
    std::string            name;
    std::vector<FieldInfo> fields;
    bool                   is_class;
};

struct FieldInfo {
    std::string name;
    std::string type;
    bool        is_pub;
};
```

При инициализации структуры `Point { x: 1, y: 2 }` анализатор:
1. Ищет `Point` в `type_table_`
2. Проверяет наличие каждого поля
3. Проверяет доступность (pub)
4. Проверяет совместимость типов значений

## Таблица данных для LSP

`SemanticAnalyzer` предоставляет геттеры для внешних инструментов:

```cpp
const FuncTable& func_table() const { return func_table_; }
const std::unordered_map<std::string, TypeInfo>& type_table() const { return type_table_; }
```

Автодополнение (`complete.cpp`) и Language Server (`lsp/server.cpp`) запускают семантический анализ и читают эти таблицы для формирования подсказок.
